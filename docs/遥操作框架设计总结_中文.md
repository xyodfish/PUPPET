# 新一代机器人遥操作框架设计总结

## 一、项目命名建议

今天围绕项目命名讨论了一批候选名字，目标是：

- 实用
- 好记
- 有一点趣味性
- 最好能和遥操作、映射、接管、分身、编排这些语义对应起来
- 如果可以，名字还能反向展开成一组有意义的英文单词

### 当前比较强的候选

#### 1. PUPPET
这是目前最贴合“遥操作”语义的名字。

可展开为：

- **P**rimitive-based
- **U**nified
- **P**latform for
- **P**luggable
- **E**mbodied
- **T**eleoperation

它的优点：

- 和遥操作天然有关联，“提线木偶”的感觉很强
- 有记忆点
- 有趣，但不至于太轻浮
- 很适合表达“人通过多种输入方式操控远端机器人”的感觉
- 也能对应这套框架里“统一 Primitive、中间层、Pipeline、Retargeting”的设计思路

#### 2. WARP
这个名字更偏“跨空间映射”和“远程投射”的感觉。

可展开为：

- **W**orkspace-aware
- **A**bstraction and
- **R**etargeting
- **P**latform

它的优点：

- 短
- 很像一个成熟工程系统名字
- 很适合表达“从人的动作空间到机器人动作空间”的映射

#### 3. PILOT
这个名字偏“操控员/驾驶员”的感觉。

可展开为：

- **P**luggable
- **I**nput and
- **L**imb
- **O**rchestration for
- **T**eleoperation

它的优点：

- 很自然
- 工程感比较强
- 容易对外解释

### 当前结论

如果希望名字：

- 和遥操作最贴
- 最有记忆点
- 还带一点趣味性

那么目前最推荐的是：

# **PUPPET**

---

## 二、这套系统的定位

这不是一个单纯的“遥操作功能模块”，而是一套面向多关节机器人的**通用遥操作运行框架**。

它需要支持：

- 多种异构输入设备和传感器
- 多种 Retargeting 策略
- 多种输出形式
- 与机器人模型相关的约束求解后端
- 后续继续扩展到 IK、TSID / WBC、优化、Self-collision、Environment collision 等能力

在当前阶段，通信细节和最终 Target 的传输格式不是设计中心。  
当前真正的设计中心，是：

> **输入设备到机器人可执行控制之间的内部结构设计**

---

## 三、核心设计思想

这套系统里最重要的抽象，不是通信，也不是某个具体机器人，而是：

# **统一的中间控制语义层**

也就是：

# **PrimitiveFrame**

整体主链路可以概括为：

```text
Device / Source
  -> PrimitiveFrame
  -> Orchestrator
  -> Retargeting Pipeline
  -> Robot-Aware Control Intent
  -> Robot Backend
  -> Final Target
```

也就是说：

- Device / Source 负责提供输入
- PrimitiveFrame 负责统一表达这些输入
- Orchestrator 负责管理和编排
- Retargeting Pipeline 负责把输入变成更接近机器人控制的中间意图
- Robot Backend 负责结合机器人模型、任务和约束做求解
- 最后再生成 Final Target

---

## 四、为什么必须引入 Primitive 层

如果每个设备都直接输出机器人 Target，很快就会出现严重耦合：

- VR 会绑定某一类手臂控制方式
- 主从臂会绑定某一个特定从臂
- 动捕很难自然接入
- 视频算法模块难以复用
- 多来源融合会越来越混乱
- Python 算法模块会更难和 C++ 主系统解耦

因此，这套框架必须引入一个稳定的中间层。  
这一层不是：

- 设备原始数据
- 也不是最终机器人 Target

而是：

> **带有控制语义、但还没有绑定具体机器人执行细节的 Primitive 层**

---

## 五、为什么名字从 Bundle 改成 PrimitiveFrame

在讨论 protobuf 的时候，原来用过 `Bundle` 这个名字。  
后面决定换掉，原因是：

- `Bundle` 太泛
- 不能很好表达“同一时刻的一帧输入语义”
- 在遥操作系统里，不如 `Frame` 直观

所以最终更推荐：

# `PrimitiveFrame`

这个名字的好处：

- 更像一帧连续控制流
- 更适合缓存、调试、日志和主循环
- 很自然地表达“某个 Source 在某个时刻给出的一组 Primitive”

典型用法：

- latest primitive frame
- mocap primitive frame
- source frame cache
- processFrame()

---

## 六、Primitive 的第一版建议类型

为了避免第一版设计过胖，Primitive 的类型先收敛到以下几类：

- `PosePrimitive`
- `TwistPrimitive`
- `JointStatePrimitive`
- `JointCommandPrimitive`
- `ScalarPrimitive`
- `BooleanPrimitive`
- `ModePrimitive`
- `PlanarMotionPrimitive`
- `SkeletonPrimitive`
- `LandmarkSetPrimitive`

### 各类 Primitive 的含义

#### 1. PosePrimitive
表示某个控制实体的位姿参考。

例如：

- 左手控制器 Pose
- 右手控制器 Pose
- Head Pose
- 手腕 Pose
- 骨盆 Pose

#### 2. TwistPrimitive
表示某个控制实体的速度参考。

例如：

- 手臂末端的 Cartesian velocity
- 头部旋转速度
- 某个控制点的线速度和角速度

#### 3. JointStatePrimitive
表示关节的观测状态。

例如：

- 主手关节状态
- 手套关节状态
- 某个输入设备的关节状态

#### 4. JointCommandPrimitive
表示关节命令参考。

例如：

- Position command
- Velocity command
- Effort command

第一版用一个统一的 `JointCommandPrimitive`，先不急着拆得太细。

#### 5. ScalarPrimitive
表示一个标量控制量。

例如：

- grip
- trigger
- openness
- speed scale

#### 6. BooleanPrimitive
表示开关类状态。

例如：

- clutch
- enable
- freeze
- deadman

#### 7. ModePrimitive
表示模式切换。

例如：

- control mode
- teleop mode
- retargeting mode

#### 8. PlanarMotionPrimitive
表示底盘平面运动意图。

例如：

- vx
- vy
- wz

#### 9. SkeletonPrimitive
表示整个人体骨架。

这对于动捕和视频输入特别重要。

#### 10. LandmarkSetPrimitive
表示关键点集合。

例如：

- hand landmarks
- upper body landmarks
- face landmarks

---

## 七、PrimitiveFrame 的 protobuf 设计结论

### 顶层设计结论

不要使用顶层的 `oneof Primitive` 包装方式。

原因是：

- 同一个时刻可能同时有多种 Primitive 并存
- 一个 Source 一帧里往往会同时输出多类 Primitive
- C++ Manager 更适合按类型批量处理
- Python 节点也更适合直接构造 typed repeated fields

因此更合适的方式是：

一个 `PrimitiveFrame` 中直接按类型并列放置：

- repeated poses
- repeated twists
- repeated joint_states
- repeated joint_commands
- repeated scalars
- repeated booleans
- repeated modes
- repeated planar_motions
- repeated skeletons
- repeated landmark_sets

### 一个 PrimitiveFrame 的语义边界

建议约束为：

- 一个 Source
- 一个时间片
- 一个语义上下文

不要把多个不相关的 Source 强行塞进同一个 PrimitiveFrame。

---

## 八、设备层的职责

设备层不应该直接输出最终机器人 Target。

它更合适的职责是：

> **把设备原始输入转换成统一的 PrimitiveFrame**

### 例子

#### VR Source 可能输出
- 左右控制器 Pose
- Head Pose
- trigger 值
- clutch 状态
- 当前 mode

#### 主从臂 Source 可能输出
- 主手关节状态
- 末端 Pose
- gripper 开合量

#### 动捕 Source 可能输出
- 全身 Skeleton
- 手腕 Pose
- Head Pose
- 手部 Landmark
- hand openness

---

## 九、动捕 / 视频接入的统一思路

动捕和视频输入不能简单看成“直接给机器人 Target 的设备”。  
它们更像是：

> **提供人体状态估计结果的输入源**

因此，接入动捕 / 视频时更合理的主链路是：

```text
Mocap / Vision Source
  -> SkeletonPrimitive / LandmarkSetPrimitive
  -> Human Feature Extraction
  -> 更高质量的控制 Primitive
  -> Retargeting Pipeline
  -> Robot-Aware Control Intent
```

### 为什么这样更合理

因为动捕 / 视频输入通常具有以下特点：

- 更像人体状态估计，而不是直接控制命令
- 经常是部分可见、部分丢失
- 每个身体部位的 confidence 可能不同
- 后续需要做归一化、语义提取和筛选

所以不要强行要求动捕直接输出最终机器人 Target。

更好的方法是：

- 先输出 Skeleton / Landmark
- 再通过 Feature Extraction 变成更适合 Retargeting 的 Primitive
- 再进入统一框架

---

## 十、C++ 与 Python 混合架构的结论

### 总体结论

应该采用：

- **C++ 作为主框架语言**
- **Python 只在需要时作为外部算法节点**

### 不建议的做法

- 不建议把整个 Manager 用 Python 写
- 不建议让 Python 成为主编排语言
- 不建议把 Python 大量嵌入 C++ 主进程，作为默认架构

### 为什么

因为主框架仍然需要稳定承担：

- 生命周期管理
- Source 管理
- body-group 级编排
- 模式切换
- Target 组装
- 与机器人后端的稳定集成

而 Python 最适合承载的是：

- 动捕处理
- 视频 Pose / hand / gesture 模块
- 基于开源方案的快速试验
- 模型推理和感知侧的处理逻辑

因此更准确的定位应该是：

> **Python 模块是外部算法节点，而不是系统骨架**

---

## 十一、如何理解 PrimitiveFrame in / PrimitiveFrame out

这个概念的含义不是：

“所有模块永远都只能在 Primitive 层工作”。

真正的意思是：

> **在模块边界上，PrimitiveFrame 是一个稳定的跨语言、跨进程契约**

### 例子 1：mocap feature node
输入：
- SkeletonPrimitive

输出：
- Head PosePrimitive
- 左右手腕 PosePrimitive
- hand openness ScalarPrimitive

### 例子 2：gesture node
输入：
- LandmarkSetPrimitive

输出：
- BooleanPrimitive
- ScalarPrimitive
- ModePrimitive

### 例子 3：native retargeting helper
输入：
- PosePrimitive

输出：
- 更接近机器人控制语义的 Pose / Joint Command

### 需要注意的一点

模块内部不一定要一直抱着 protobuf 对象计算。  
模块内部完全可以使用自己更高效的结构体和数据结构。  
`PrimitiveFrame` 的主要作用是：

- 作为模块边界协议
- 方便 C++ / Python 混合
- 方便多进程通信
- 方便调试与记录

---

## 十二、主运行时的总体结构

推荐的总体运行方式是：

- 一个 C++ 主进程
- 若干可选的外部 Source 进程
- 若干可选的外部 Python 算法进程

这是一个：

> **核心集中、边缘解耦**

的结构。

### 主进程负责什么

主进程应当掌握整个系统的控制权，负责：

- Source 的收集和 freshness 管理
- 运行时 mode 的解析
- body-group ownership 和仲裁
- Retargeting Pipeline 的调度
- Native C++ Retargeting 节点的运行
- 外部 Python 节点结果的接入
- Final Target 的组装
- 运行状态发布
- 诊断与录制

### 核心原则

主进程应当始终是以下事情的唯一权威：

- 当前哪个 Source 控哪个 body group
- 当前系统处于什么 mode
- 哪些输入有效
- 如何融合多个来源
- 如何形成最终 Target

---

## 十三、建议的进程拓扑

### 必需进程
- `teleop_runtime`（C++ 主进程）

### 可选 Source 进程
- `vr_device_node`
- `master_arm_node`
- `mocap_adapter_node`
- `joystick_node`

### 可选算法进程
- `python_mocap_feature_worker`
- `python_video_pose_worker`
- `python_gesture_worker`

### 可选工具进程
- `teleop_recorder`
- `teleop_visualizer`
- `teleop_supervisor`

### 设计原则

外部进程只负责提供能力。  
系统的编排控制权必须始终留在 C++ 主进程。

---

## 十四、主进程的主循环理解

主进程的主链路可以概括为：

```text
Collect source frames
  -> refresh source cache and health
  -> resolve mode / ownership / routing
  -> build per-group input
  -> run retargeting pipelines
  -> merge control intents
  -> run robot backend
  -> build final target
  -> publish / record / monitor
```

换成更通俗的话说，就是：

1. 收集各个 Source 的 PrimitiveFrame
2. 更新缓存、超时和健康状态
3. 决定当前谁控制哪个 group
4. 为每个 group 组装输入
5. 跑对应的 Retargeting Pipeline
6. 合并不同 group 的控制意图
7. 进入机器人后端求解
8. 生成最终 Target
9. 发布、记录和监控

## 十六、为什么必须增加 control 层

在 Primitive 和 Final Target 之间，还需要一个更接近机器人控制语义的中间层：

# `control/`

建议在这一层定义：

- `ControlIntent`
- `GroupControlIntent`
- `CartPoseIntent`
- `JointCommandIntent`
- `PostureIntent`
- `BaseMotionIntent`
- `ConstraintRequest`

### 各层语义区别

#### Primitive 层
表达设备/输入语义

#### Control 层
表达机器人控制语义

#### Robot Backend 层
表达模型、任务、约束和求解语义

#### Target 层
表达最终下发语义

这个分层对后续演进非常重要。

---

## 十七、为什么必须增加 robot backend 层

一旦系统要支持：

- IK
- TSID / WBC
- 优化后端
- joint / velocity / acceleration limit
- self-collision
- environment collision
- CoM / support 约束

这些能力就不能继续散落在各个 Retargeting 模块里。

它们必须上升成显式的：

# `robot/` backend 层

### 推荐的子目录

- `robot/model/`
- `robot/kinematics/`
- `robot/dynamics/`
- `robot/geometry/`
- `robot/constraints/`
- `robot/tasks/`
- `robot/solvers/`
- `robot/backend/`

### 核心思想

Retargeting 负责表达：

> 想做什么

Robot Backend 负责决定：

> 在模型和约束下，怎样做出来

---

## 十八、Robot Backend 的建议抽象

建议支持以下几类后端：

- `DirectMappingBackend`
- `IkBackend`
- `TsidBackend`
- `OptimizationBackend`

运行时应支持：

- 按 group 选择 backend
- 按 mode 选择 backend
- 在某些场景下使用 whole-body backend

### 例子

- arm 在 direct mode 下，用 `DirectMappingBackend`
- arm 在 pose mode 下，用 `IkBackend`
- upper body whole-body teleop，用 `TsidBackend`
- 更复杂的安全受限 whole-body mode，用 `OptimizationBackend`

---

## 十九、Task 与 Constraint 的区分

这两个概念必须分开。

### Task = 想实现的目标
例如：

- End-effector pose tracking
- Joint tracking
- posture regularization
- CoM tracking

### Constraint = 必须满足的条件
例如：

- joint bounds
- velocity bounds
- acceleration bounds
- self-collision avoidance
- environment collision avoidance
- support constraints

这层区分清楚后，后续 Backend 设计会干净很多。

---

## 二十、碰撞与几何处理的建议链路

关于 Self-collision 和 Environment collision，建议采用如下结构：

```text
Geometry model
  -> collision / distance query
  -> collision feature extraction
  -> constraint builder
  -> backend solver
```

这样做的好处是：

- 运行时 safety check 可以复用
- 约束构造可以复用
- 更容易接 Pinocchio / hpp-fcl
- 后续也更容易接环境碰撞

---

## 二十一、建议的开发顺序

### 第一阶段
先搭起：

- Primitive 层
- Source Manager
- Orchestrator
- Retargeting Manager
- Target Builder
- 简单 Native Backend

### 第二阶段
补上：

- 动捕 / 视频输入
- Python 外部节点支持
- Human Feature Extraction

### 第三阶段
补上：

- Robot Model 层
- IK Backend
- Collision Interface
- Constraint Manager

### 第四阶段
补上：

- TSID / Optimization Backend
- Self-collision Constraint
- whole-body reactive control 特性

---

## 二十二、最后的整体设计陈述

这套新仓库最终应当被建设成：

> 一个以 C++ 为中心的遥操作运行框架。它通过统一的 `PrimitiveFrame` 中间层接入异构输入，使用 body-group 级编排和可组合的 Retargeting Pipeline 生成机器人控制意图，并通过可选的 Python 外部算法节点和面向模型约束的 Robot Backend，最终形成可执行的控制目标。

---

如果后面写 README，可以用下面这段作为初版介绍：

```text
PUPPET 是一套面向多关节机器人的遥操作运行框架。它通过统一的 PrimitiveFrame 表达多源输入，使用可配置的 Retargeting Pipeline 和 body-group 级编排生成机器人控制意图，并可结合基于模型的后端求解能力，最终输出可执行的控制目标。
```
